SMERP Authentication Protocol v1

1. Overview
===========

SMERP Authentication provides proof-of-possession of a client's long-term
X25519 private key.

The client and relay each hold an X25519 keypair. Authentication is performed
by deriving a shared secret and creating an HMAC signature over a timestamp.

The protocol is stateless and consists of a single HTTP request.

Usage:

    GET /envelopes

2. Terminology
==============

Client
    Entity requesting authentication.

Relay
    Entity verifying authentication.

PKH
    Hex-encoded X25519 public key.

Timestamp
    Unix timestamp in seconds encoded as a decimal UTF-8 string.

Signature
    Hex-encoded HMAC-SHA256 output.


3. Cryptographic Primitives
===========================

Shared secret:

    shared_secret =
        X25519(
            private_key,
            public_key
        )

Proof key:

    proof_key =
        SHA256(shared_secret)

Signature:

    signature =
        HMAC-SHA256(
            proof_key,
            UTF8(timestamp)
        )

All cryptographic operations use raw bytes.
Hex encoding is only used for transport.


4. Timestamp Rules
==================

The timestamp MUST:

- be a Unix timestamp in seconds
- be represented as a base-10 decimal string
- contain no leading zeros
- be encoded as UTF-8 before signing


Example:

    1750201234


The relay defines the accepted clock-skew window.
Requests outside this window MUST be rejected.


5. Client Authentication
========================

The client computes:

    shared_secret =
        X25519(
            client_private_key,
            relay_public_key
        )


    proof_key =
        SHA256(shared_secret)


    signature =
        HMAC-SHA256(
            proof_key,
            UTF8(timestamp)
        )


The client sends:

    GET /envelopes


Headers:

    x-smerp-timestamp: <timestamp>
    x-smerp-signature: <signature_hex>


Requirements:

    x-smerp-timestamp
        MUST contain the signed timestamp.

    x-smerp-signature
        MUST contain lowercase hexadecimal encoding of
        the HMAC-SHA256 output.


6. Relay Verification
=====================

The relay extracts:

    pkh (client public key hex)
    timestamp
    signature


The relay MUST:


1. Validate the timestamp.

   Requests outside the accepted clock-skew window MUST be rejected.


2. Compute:

    shared_secret =
        X25519(
            relay_private_key,
            client_pkh
        )


    proof_key =
        SHA256(shared_secret)


    expected_signature =
        HMAC-SHA256(
            proof_key,
            UTF8(timestamp)
        )


3. Compare:

    expected_signature

against:

    received_signature


The comparison MUST use a constant-time comparison function.


If the signatures match, authentication succeeds.
Otherwise authentication fails.


7. Security Properties
======================

Provides:

- Proof-of-possession of the client's X25519 private key.
- Stateless relay verification.
- Replay resistance bounded by the timestamp acceptance window.


Does not provide:

- Single-use authentication proofs.
- Protection against replay within the accepted timestamp window.
- Forward secrecy for authentication requests.


TLS SHOULD be used for transport protection.
