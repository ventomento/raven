SMERP Authentication Protocol v1

1. Overview
===========

SMERP Authentication Protocol v1 provides proof-of-possession of a client's
X25519 private key.

The client and relay each possess a long-term X25519 keypair.
Authentication is performed by deriving a shared secret using X25519 and
producing an HMAC over a timestamp.

The protocol requires a single request and no server-side session state.


2. Terminology
==============

Client
    Entity performing authentication.

Relay
    Server verifying authentication.

PKH
    Hex-encoded X25519 public key.

Timestamp
    Unix timestamp in seconds.

Shared Secret
    X25519 Diffie-Hellman output.

Proof Key
    SHA-512 hash of the shared secret.

Signature
    HMAC-SHA-512 authentication tag.


3. Cryptographic Primitives
===========================

Shared Secret Derivation

    shared_secret =
        X25519(
            private_key,
            public_key
        )

Proof Key Derivation

    proof_key =
        SHA512(shared_secret)

Signature Generation

    signature =
        HMAC_SHA512(
            proof_key,
            message
        )


4. Timestamp Encoding
=====================

The timestamp MUST be a Unix timestamp in seconds.

The timestamp MUST be represented as a base-10 decimal string encoded as
UTF-8.

Example:

    1750201234

No leading zeros are permitted.


5. Client Procedure
===================

5.1 Shared Secret

The client computes:

    shared_secret =
        X25519(
            client_private_key,
            relay_public_key
        )

5.2 Proof Key

The client computes:

    proof_key =
        SHA512(shared_secret)

5.3 Signature

The client computes:

    message =
        UTF8(timestamp)

    signature =
        HMAC_SHA512(
            proof_key,
            message
        )

5.4 Request Headers

The client MUST include the following HTTP headers:

    x-smerp-pkh: <client_pkh>
    x-smerp-timestamp: <timestamp>
    x-smerp-signature: <signature_hex>

Where:

    client_pkh
        Hex-encoded X25519 public key.

    timestamp
        Unix timestamp in seconds.

    signature_hex
        Lowercase hexadecimal encoding of the HMAC output.


6. Relay Verification
=====================

6.1 Extract Inputs

The relay extracts:

    client_pkh
    timestamp
    signature

from the request headers.

6.2 Timestamp Validation

The relay MUST verify that the timestamp falls within its accepted
clock-skew window.

Requests outside the accepted window MUST be rejected.

The size of the acceptance window is implementation-defined.

6.3 Shared Secret

The relay computes:

    shared_secret =
        X25519(
            relay_private_key,
            client_pkh
        )

6.4 Proof Key

The relay computes:

    proof_key =
        SHA512(shared_secret)

6.5 Expected Signature

The relay computes:

    message =
        UTF8(timestamp)

    expected_signature =
        HMAC_SHA512(
            proof_key,
            message
        )

6.6 Verification

The relay MUST compare:

    expected_signature

and

    received_signature

using a constant-time comparison function.

If the signatures match, authentication succeeds.

Otherwise, authentication fails.


7. Security Properties
======================

This protocol provides:

    - Proof-of-possession of the client's X25519 private key.
    - Stateless relay verification.
    - Single-request authentication.
    - Replay resistance bounded by the relay's timestamp acceptance window.

This protocol does not provide:

    - Single-use authentication proofs.
    - Forward secrecy for authentication proofs.
    - Protection against replay within the accepted timestamp window.

Deployments SHOULD use TLS for transport security.
